FEATURE FACTORY
Reten Platform · Nueva forma de trabajar

Feature
Factory

Un pipeline de 6 fases que transforma un requerimiento de negocio ambiguo en código implementado, revisado y validado — orquestado por agentes de IA especializados con feedback loop integrado.

¿Por qué esto cambia todo? Antes, cada dev decidía cómo abordar una feature. Ahora el proceso mismo es el producto: discovery estructurado, specs técnicos por repo, gate de arquitectura obligatorio, implementación paralela en worktrees aislados, y un feedback loop que convierte errores en reglas preventivas.
🎯
Single Point of Contact
El usuario solo habla con el Orchestrator. Toda pregunta de sub-agentes fluye a través de él — nunca directamente al usuario.
🔌
Skills Dinámicas
Cada agente carga skills según el repo donde opera. El mismo Tech Lead se comporta diferente en reten-api que en reten-app.
🧑‍💻
Human-in-the-loop Selectivo
Solo preguntas de negocio y decisiones arquitectónicas críticas escalan al usuario. El resto lo resuelven los agentes.
📄
Artefactos Persistentes
Cada fase produce documentos .md que sirven de input a la siguiente. Todo el razonamiento queda trazable en el repo.
⚡
Paralelismo Inteligente
El TL de cada repo decide: secuencial, paralelo por repo, o máximo paralelismo en worktrees aislados por slice.
🚦
Architecture Gate
Ningún código se escribe sin pasar por revisión de Clean Architecture, SOLID y Security Audit completo.
🔄
Feedback Loop
Cada ejecución registra incidentes. La retrospectiva extrae pitfalls y los promueve a CLAUDE.md — el pipeline mejora con cada feature.

Agentes

Cada agente es un rol con skills inyectadas dinámicamente según el contexto y el repo.

🎯
Orchestrator
Siempre activo
Rol
Director de orquesta. Única interfaz con el usuario.
Skills
Ninguna — conoce el workflow y delega.
Decide
Qué repos involucrar, cuándo escalar, cuándo avanzar de fase.
🔍
Business Analyst
Phase 1 · /discover
Rol
Product analyst que desafía y enriquece el requerimiento.
Estilo
2–4 preguntas por ronda. Desafía supuestos. Nunca asume.
Output
business-spec.md
⚙️
Tech Lead
Phase 2 · 1 por repo involucrado
Rol
Escanea el codebase, identifica impacto, delega a specialists.
Skills
Carga dinámica según el repo (ver sección Skills).
Sub-agentes
DB Spec, Endpoints Spec, UI Spec, State Spec, Senior Dev
🏗️
Plan Architect
Phase 3 · /plan-feature
Rol
Consolida todos los specs en un plan ejecutable con stages y slices.
Sizing
Small <15 · Medium 15–40 · Large 40+ con slices
Output
FEATURE_PLAN.md + stages/
🚦
Architecture Reviewer
Phase 4 · GATE obligatorio
Revisa
Clean Architecture, SOLID, Design Patterns, DRY, Security, Multi-tenancy
Puede
Auto-corregir (low) · Sugerir (medium) · Bloquear → P3 (high/critical)
Sub-agente
Security Auditor (pase dedicado)
💻
Implementation Agents
Phase 5 · worktrees aislados
Rol
Ejecutan el plan aprobado produciendo código + tests.
Post-impl
sync-api-contract → Security Validation → Code Review
Paralel.
El TL decide: secuencial / paralelo por repo / máximo
🔄
Retrospective Analyst
Phase 6 · /retrospective
Rol
Analiza incidentes, extrae pitfalls sistémicos, propone mejoras a CLAUDE.md.
Input
incidents.md + review-report.md + tech specs + git history
Output
retrospective.md + pitfalls/PIT-*.md + CLAUDE.md updates
